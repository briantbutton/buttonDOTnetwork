The Case for ButtonNet

Foundation Belief

There are some situations where a local server is best for sharing digital content.  This is strongest when:
       ●  More than two interested parties under one roof
       ●  Shared content is large
       ●  There is immediate interest in the content

A local server becomes the only plausible option with these further factors. 
       ●  Multiple parties are generating the content
       ●  The Internet is unreliable, absent, or has data ceilings (yikes!)

However, the pendulum has swung to the far extreme — all networking features being delivered from the cloud.  This presents an opportunity.

Our Solution — In Overview

Local Server.  Server is a robust, fanless, black Mini-PC, manufactured in China, software imaged in California.  Remotely managed.  Almost no user setup. 

Access.  After the local server installation, a new website appears on the local WiFi with a URL like 'yeeoldesafarihouse.buttonnet.app'.  Visiting that URL in a browser on that WiFi network brings you to the ButtonNet App for your venue.
Note:  Content can be privacy protected.

UI.  In a smart phone, the experience is App-like, although the user never goes to an App store.  ButtonNet uses Google's Progressive Web App (PWA) technology.

HD viewing.  The local server has an HDMI port.  It can be used to project videos/images on an HDTV screen for collective viewing.

Portable.  The local server can be taken to the field.  It runs off battery for a long time.  It hosts its own WiFi network.  Once logged in, users may continue to use ButtonNet without access to the cloud for two weeks.

Alternatives

We have heard no rumours of a comparable effort for a local media server.  (The pendulum has truly swung to the extreme.)

There is a class of network-attached storage (NAS).  NAS has been around for decades, and calls for an IT guy (or extreme geek) to set up each user.

All other alternatives are one-to-one file sharing:
       ● Airdrop   (if all parties have Apple products)    [Note 1]

Is that all?  Really??
I think so.  Cables and removeable storage devices are not plausible with current consumer products.  If you count backhaul Internet, you could add, say, Google Photos. 

~  ~  ~  ~  ~  ~  ~  ~

Let's do some math on Google Photos:     [Geek warning]
       6      Six people gather at the bar in the evening to reminisce the day's adventures
       5      Each has five good pics of other members doing something memorable.
      10M  Each pic is ten megabytes    [Note 2]
   300M  The guests must collectively upload 300 megabytes to Google photos
1,800M  The guests must collectively download 1,800 megabytes from Google photos

In a remote environment, that is an impossible ask of the backhaul Internet.  Typically guests all try at once, get timeouts and failures.  They block the other guests who are just trying to check email.  Eventually they give up.

At that moment, you realize that it is faintly ridiculous to send an image or a video to a distant server so that someone next to you may enjoy it.
